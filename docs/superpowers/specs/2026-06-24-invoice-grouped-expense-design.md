# 發票群組消費明細設計

## 目標

財政部發票匯入可能將同一張發票拆成多筆明細，其中包含正數購買品項與負數優惠券或折扣。系統應以發票號碼群組這些明細，在消費明細頁以一張發票一筆總額呈現，展開後保留每個品項與折扣資訊，並允許每個品項使用不同預算項目。

## 已確認規則

- 發票群組唯一鍵只使用 `invoice_number`（發票號碼）。
- 使用者確認資料來源不會出現同一發票號碼對應不同日期或賣方統編的情況。
- 正數明細代表購買品項。
- 負數明細代表折扣、優惠券或折扣碼。
- 發票實付總額為同一發票號碼所有明細金額的正負加總。
- 每個正數品項可以指定不同預算項目。
- 同一發票的所有負數折扣先加總，再歸入原始金額最高的正數品項。
- 若最高金額相同，折扣歸入匯入順序最前面的正數品項。
- 非發票來源的手動消費維持目前單筆顯示。

## 資料模型

### 發票號碼

在 `invoice_drafts` 與 `expenses` 加入明確的 `invoice_number` 欄位。匯入時直接從財政部 CSV 的「發票號碼」取得，不再要求前端從 `source_line_key` 反向解析。

`source_line_key` 仍保留做明細列層級的重複檢查。發票號碼負責群組，兩者用途不得混用：

- `invoice_number`：判斷哪些明細屬於同一張發票。
- `source_line_key`：判斷同一張發票中的特定明細是否已匯入。

### 消費資料粒度

資料庫維持每個品項一筆 `expenses` 紀錄，不新增發票主表。這能延續現有預算統計、品項編輯與分類流程。

每筆由發票確認產生的 `expenses` 紀錄保存：

- `invoice_number`
- 原始品項名稱
- 原始明細金額
- 折扣分配後的記帳金額
- 預算項目
- 支付工具與信用卡
- 原始明細識別 `source_row_id` / `source_line_key` 對應資訊

為避免失去稽核能力，新增 `original_amount` 保存匯入原始金額，既有 `amount` 保存折扣分配後、實際計入預算與付款排程的金額。

### 折扣明細保存

負數折扣不能在確認後消失。系統建立對應的明細資料，並以新增的 `line_type` 標記：

- `item`：正數購買品項。
- `discount`：負數折扣。

折扣列保留原始負數金額與品名，但不另外建立付款排程或重複計入預算。其金額會反映在被分配品項的 `amount` 中。

## 匯入與確認流程

1. CSV 解析每一列時讀取發票號碼並寫入 `invoice_drafts.invoice_number`。
2. 每列仍產生獨立 `source_line_key`，維持現有重複檢查。
3. 待確認頁依 `invoice_number` 群組顯示發票。
4. 群組內正數列可分別選擇預算項目。
5. 支付工具、信用卡與分期期數屬於整張發票，群組內共用，不允許各品項使用不同支付工具。
6. 確認時先找出金額最高的正數品項；相同金額時採匯入順序最前者。
7. 將所有負數折扣加總，套用到該品項的記帳金額。
8. 建立各正數品項的 `expenses` 紀錄及折扣明細關聯資料。
9. 整張發票只依實付總額建立一次付款效果，避免每個品項各自造成帳單或現金流重複計算。
10. 所有 draft 成功處理後才統一標記為 `confirmed`。

## 付款排程

目前每筆 expense 都會建立付款排程，但同張發票共用支付工具且可能分期。為避免不同品項各自四捨五入或產生多組分期排程，付款排程應以整張發票實付總額為基礎。

確定採用以下做法：

- 選定一筆發票主付款 expense，預設使用金額最高的正數品項。
- 該 expense 建立整張發票的付款排程。
- 其餘同發票品項保留預算金額，但不建立獨立付款排程。
- 在 expenses 加入可空白的 `payment_parent_expense_id`，子品項指向負責付款排程的主品項。

這樣預算仍按品項統計，信用卡帳單與現金流則只計算一次發票實付總額。

## 消費明細介面

### 收合狀態

發票來源預設顯示一列群組摘要：

- 消費日期
- 店家
- 發票號碼
- 正數品項數量
- 折扣總額（有折扣時顯示）
- 實付總額
- 支付方式與信用卡
- 展開按鈕

手動消費維持既有列顯示。

### 展開狀態

展開後依原始匯入順序顯示：

- 品項名稱
- 原始金額
- 折扣列與負數金額
- 折扣分配後金額
- 預算項目
- 編輯與刪除操作

被分配折扣的品項應清楚顯示，例如「原價 82、折扣 -1、計入 81」。

### 編輯與刪除

- 修改品項名稱或預算項目不影響發票實付總額。
- 修改正數品項金額或折扣金額後，重新計算最高金額品項、折扣歸屬及發票總額。
- 刪除目前承接折扣的品項後，折扣自動重新分配給剩餘最高金額品項。
- 刪除發票最後一個正數品項時，應阻止操作或要求刪除整張發票。
- 若發票只剩折扣而沒有正數品項，標記為資料異常，不建立正式消費。

## 舊資料處理

既有 expenses 未保存獨立 `invoice_number`，不能可靠地只靠店家與日期回推。因此分兩階段：

1. 新匯入資料立即採用新結構。
2. 既有資料僅在能從 `source_row_id` 或關聯 `invoice_drafts.source_line_key` 明確取得發票號碼時回填；無法確認者維持舊單筆顯示，不猜測合併。

資料遷移不得因同日同店家而自動合併。

## 重複檢查

- 同一發票號碼可以有多筆品項，不能只因發票號碼已存在就跳過整張發票。
- 每一列仍以 `source_line_key` 判斷是否重複。
- 已確認的明細與 active expense 阻止相同明細重複匯入。
- 同發票新增一個未見過的明細列時，只匯入新列，不重建既有列。

## 錯誤處理與一致性

確認整張發票必須視為單一資料庫交易。API 透過 PostgreSQL RPC 執行群組確認，避免目前 REST 多次寫入造成部分成功。交易規則如下：

- 任一品項建立、折扣分配、付款排程或 draft 更新失敗時，整個 RPC transaction rollback，不得留下部分 confirmed、部分 pending 的狀態。
- API 回傳發票號碼與失敗階段，方便前端顯示可理解的錯誤。
- 發票總額、正數品項折扣後合計、付款排程合計三者必須完全相等。
- 若折扣絕對值大於最高品項金額，允許該品項折扣後為負數前必須先阻止確認並要求人工處理，避免負數正式消費破壞預算與付款排程。

## 測試要求

### 純函式測試

- 同發票多筆正數品項正確群組。
- 正負金額正確加總為實付總額。
- 多筆折扣加總後歸入最高金額品項。
- 最高金額相同時選擇最先匯入品項。
- 折扣重新分配後，各品項合計等於發票總額。
- 發票號碼相同但 `source_line_key` 不同的明細不被錯誤去重。
- 重複 `source_line_key` 仍被阻止。

### API 測試

- 確認整張發票後建立多筆品項 expense，但只建立一組付款排程。
- 每筆 expense 保存發票號碼與原始金額。
- 折扣列保留且不重複計入預算或付款排程。
- 任一步驟失敗時不留下部分確認結果。

### 畫面測試

- 消費明細預設每張發票只顯示一列。
- 展開後能看到所有品項與折扣。
- 每個正數品項可獨立修改預算項目。
- 修改或刪除後總額即時更新。
- 手動消費不受發票群組功能影響。

## 驗收條件

- 同發票號碼的所有明細在消費明細只佔一個群組摘要。
- 群組實付總額等於原始正負明細加總。
- 每個正數品項可以使用不同預算項目。
- 所有折扣歸入最高金額正數品項；同額時使用最先匯入品項。
- 預算統計、信用卡帳單與現金流均不重複計算。
- 發票品項、折扣與原始金額可追溯。
- 既有無法可靠辨識發票號碼的消費不被錯誤合併。
